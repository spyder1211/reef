# Table++ v0.3 改善提案 — アプリ全体監査レポート

> 作成日: 2026-06-13 / 対象: v0.2.0（main `afb5805` 時点）/ 目的: v0.3 で着手すべき改善点の洗い出しと優先度付け
>
> 本書はコードの実読（main / preload / renderer / 設定一式）と、過去の意図的後回し項目（プロジェクトメモリ）を突き合わせて作成しました。**最終的な v0.3 スコープの決定は開発者（あなた）の判断**です。本書は判断材料としての提案です。

## 0. エグゼクティブサマリ

v0.2.0 で「開発者向け機能強化」（スキーマ表示・SQL補完・履歴・EXPLAIN・SSHトンネル・**本番接続ガード**）を出し、**本番/ステージングへ接続して書き込む**ユースケースに踏み込みました。一方でアプリ全体を見渡すと、その新しい責務を支える土台にいくつか構造的な穴があります。

最重要は次の3点です。

1. **本番ガードが「見せかけ」になっている** — 接続時の確認ダイアログと赤バーは renderer だけに存在し、main プロセスは「今 production に繋いでいるか」を一切知りません（`ConnectionConfig` に `tag` が無い）。そのため **SQLタブの任意実行・DROP/TRUNCATE・ダンプ import が本番でも無確認でバイパス可能**。とりわけ import はネイティブ File メニューから起動するので、renderer の confirm を物理的に通りません。
2. **長時間クエリをキャンセルできない** — `KILL QUERY` も `AbortController` も無く、重いクエリを流すとそのタブが完了まで固着し中断不能。DB クライアントとして致命的な欠落です。
3. **大規模結果でフリーズし得る** — 行の仮想スクロールが無く全行 DOM 化、かつ SQLタブの素の `SELECT` に自動 LIMIT が付かず全件をメモリにバッファ→全件転送→全件描画します。

加えて、**CI / Lint / 自動更新 / コード署名が一切無く**、レンダラーコンポーネントのテストもゼロです。機能追加の前に「壊れた変更を弾く土台」を整えると、v0.3 以降の開発速度と安全性が上がります。

## 1. 安全性・セキュリティ（最優先カテゴリ）

> v0.2.0 で本番接続・SSH・ダンプ入出力を入れた以上、ここが v0.3 の主役になるべきです。なお **SQLインジェクション対策（値のプレースホルダ化・識別子のバッククォート2重化・演算子/LIMITホワイトリスト）は一貫して堅実**で、ここは良くできています（`filterBuilder.ts` / `editBuilder.ts` / `sqlDumpHelpers.ts`）。

### S1【高・中工数】本番ガードが main 側に届かず、破壊的操作をブロックできない ★最重要

- **現状（確認済み）**:
  - 確認ダイアログ: `src/renderer/src/store/useAppStore.ts:408`（`window.confirm`、接続時に1回だけ）
  - 警告バー: `src/renderer/src/workspace/WorkspaceShell.tsx`（赤い「PRODUCTION」表示のみ）
  - 判定: `src/renderer/src/store/helpers.ts:27`（`profile?.tag === 'production'`）
  - **`ConnectionConfig`（`src/shared/types.ts:17`）に `tag` フィールドが無く**、`getConnectConfig()` も tag を渡さない → **main プロセスは production かどうかを知らない**。
- **問題**: 本番ガードは「接続の瞬間に confirm を1回出す」＋「赤バーを出す」だけで、接続後の破壊的操作は何もブロックしません。以下がすべて**本番でも無確認でバイパス可能**:
  1. SQLタブの任意実行（`db:query`/`db:queryScript`）— `DROP DATABASE` でも `DELETE` でも追加確認なし
  2. テーブル右クリックの DROP/TRUNCATE — 通常DBと同じ1回確認のみ
  3. **SQLダンプ import/restore** — `src/main/menu.ts` のネイティブメニューから起動し `isConnected()` しか見ない。**renderer の confirm を物理的に通らず**、本番DBへ任意SQLファイルを流し込める
  4. SQLダンプ export — 本番データ全行を無確認でファイル化
- **v0.3 推奨対応**:
  - `ConnectionConfig` と main 側の接続中状態に `tag` を保持させる（`ConnectionManager` か holder に `currentTag`）。
  - production 時、`menu.ts` の import/dump に `dialog.showMessageBox` の追加確認（接続名タイプ必須など）を入れる。最低でも import は本番で強い確認 or ブロック。
  - 破壊的SQL（`DROP`/`TRUNCATE`/WHERE 無し `DELETE`・`UPDATE`）を main 側で検知して production では二段階確認。

### S2【高・小工数】safeStorage 不可時に encrypt が空文字を無言返却

- **現状（確認済み）**: `src/main/connection/createProfileStore.ts:34`
  ```ts
  encrypt(plain: string): string {
    if (!safeStorage.isEncryptionAvailable()) return ''   // ← 例外も警告も出さず空文字
    return safeStorage.encryptString(plain).toString('base64')
  }
  ```
- **問題**: 暗号化不可環境でパスワードが**無言で破棄**され、接続時に空パスワードになる。`ProfileStore.save()` の「空文字なら既存暗号文を保持」分岐と相互作用して挙動が不定に。ユーザーに暗号化が効いていないことが一切通知されない（メモリ `ui-shell-followups` の「safeStorage 注記」未実装と同根）。macOS では実害が小さいが堅牢性バグ。
- **v0.3 推奨対応**: `isEncryptionAvailable()` が false のとき、(a) フォームに「この環境ではパスワードを保存できません」を明示、(b) 平文保存するなら `plaintext:` 等でマーキングして可視化、(c) もしくは保存拒否。空文字サイレント破棄は廃止。`isEncryptionAvailable()` を IPC で renderer に渡してフォーム注記する。

### S3【高・中工数】本番CSP不在 + `sandbox: false`

- **現状（確認済み）**: `src/main/index.ts:42` で `sandbox: false`。`onHeadersReceived` による CSP は無し（`src/renderer/index.html` にコメントだけ残存）。`contextIsolation: true` / `nodeIntegration: false` は正しく設定済み（良い点）。
- **問題**: DBクライアントは「信頼できないDBの内容」を表示する前提。将来リッチ表示・Markdown・エラーHTML経路でインジェクションが生まれた場合、CSP という最後の防壁が無い。`sandbox: false` だと preload が完全な Node 権限で動くため、レンダラ侵害がメイン権限へ昇格しやすい。preload は `ipcRenderer.invoke` のラッパーのみ（`fs` 等を使っていない）ので **`sandbox: true` 化は現実的**。
- **v0.3 推奨対応**: 本番ビルドで `session.defaultSession.webRequest.onHeadersReceived` に `default-src 'self'; script-src 'self'; object-src 'none'`。dev は HMR 用に分岐。`sandbox: true` に変更し起動確認。あわせて `app.on('web-contents-created')` で `will-navigate` を自ドメイン外 deny、`setWindowOpenHandler` で全 deny（外部は `shell.openExternal`）。

### S4【中・小工数】認証情報・履歴ファイルのパーミッションが OS デフォルト

- **現状（確認済み）**: `createProfileStore.ts:26` と `QueryHistoryStore.ts` の `writeFileSync` に `mode` 指定が無く、umask 依存で多くの環境で `0644`（他ユーザー読み取り可）。`query-history.json` には実行SQL全文（値リテラルを含む可能性）が平文で残る。
- **v0.3 推奨対応**: 両ファイルの `writeFileSync` に `{ mode: 0o600 }`、既存ファイルには `chmodSync(filePath, 0o600)`。

### S5〜S8【中〜低】その他の堅牢化

- **S5 [中]** エラーメッセージのサニタイズ: `normalizeDbError.ts` が `sqlMessage` をそのまま返し、ホスト名・ユーザー名・値リテラル（`ER_DUP_ENTRY` 等）が UI に出る。SSH 失敗時も `connectWithTunnel.ts` が `err.message` を直結。画面共有・スクショ経由の漏洩面。production 接続時は要約のみ表示し詳細はログへ。
- **S6 [中]** import のサイズ上限・巨大 statement ガード不在: `SqlStatementSplitter` の `buf` が区切り `;` まで無限に伸び得る（巨大単一 statement で OOM）。gzip 展開後サイズも無制限（gzip 爆弾）。`buf` 長と展開後バイト数に上限を。
- **S7 [低]** DevTools が本番メニューにも常時露出（`menu.ts`）→ `!app.isPackaged` 時のみに。
- **S8 [低]** Electron Fuses 未設定（`RunAsNode` 無効化・`OnlyLoadAppFromAsar` 等）。`@electron/fuses` 導入で `ELECTRON_RUN_AS_NODE` 悪用を塞ぐ。SSH 秘密鍵の `readFileSync(ssh.privateKeyPath ?? '')` はサイズ/存在チェック無し（`SshTunnel.ts`）。

## 2. パフォーマンス・大規模データ

### P1【高・中工数】行の仮想スクロールが無い

- **現状（確認済み）**: `ResultsGrid.tsx` が `table.getRowModel().rows.map(...)` で全行を DOM 化。依存に `@tanstack/react-virtual` 等は**無い**。ページサイズ最大 500 行 × 列数ぶんの `<td>` を毎回生成。
- **影響**: 列の多いテーブルでスクロール・↑↓選択がもたつく。SQLタブ（自動 LIMIT なし）では数千〜数万行を一括 DOM 化してフリーズし得る。
- **v0.3 推奨対応**: `@tanstack/react-virtual` の `useVirtualizer` を導入。固定行高なら容易。ヘッダ sticky 化も併せて。

### P2【高・小工数】SQLタブの結果が全件メモリ保持 + 自動 LIMIT なし

- **現状（確認済み）**: `ConnectionManager.queryScript`/`query` は `pool.query()` のバッファモードで全行をメモリにロード。`streamRows`（ストリーミング実装）は存在するが**ダンプ専用で、クエリ経路からは呼ばれていない**。SQLエディタの素の `SELECT` に LIMIT は付かない。
- **影響**: `SELECT * FROM huge_table` で main 全行バッファ→IPC 全件転送→全件描画の三重苦。TablePlus は既定で自動 LIMIT を付けてこれを防ぐ。
- **v0.3 推奨対応**: (a) 素の `SELECT` に既定 LIMIT（設定可・例 500）を付与、または「先頭 N 行のみ表示」ガード、(b) IPC 転送行数の上限と超過警告。

### P3〜P5【中】

- **P3 [中]** CSV「全件」エクスポートが全行をメモリ展開（`exportCsv(scope:'all')` → `toCsv` で全行連結）。ダンプはストリーミングなのに非対称。main 側 `streamRows` → ファイル逐次書き出しへ寄せる。
- **P4 [中]** `buildCountQuery` の `SELECT COUNT(*)` をフィルタ/初回表示のたびに実行（巨大 InnoDB で遅い）。COUNT を非同期化して行表示を先行、または概算件数（`information_schema.TABLES.TABLE_ROWS`）オプション。深い OFFSET はキーセットページネーション検討。
- **P5 [中]** `mysql2` のクエリ/接続タイムアウト未設定（`createPool` に指定なし）。`connectTimeout` と文単位タイムアウトを設定可能に。

## 3. UX・操作性・アクセシビリティ

### U1【高・中工数】長時間クエリをキャンセルできない ★最重要級

- **現状（確認済み）**: `cancel`/`abort`/`KILL QUERY` の仕組みが無い。`runSql`/`runTable` は `await window.api.query(...)` を投げっぱなしで、実行中は「実行中…」表示とボタン disabled になるだけ。
- **影響**: 重いクエリを流すとタブが完了（または接続断）まで固着。Cmd+R も効かず接続を切るしかない。
- **v0.3 推奨対応**: 専用接続でクエリを走らせ `SELECT CONNECTION_ID()` を保持。実行中に出す「停止」ボタンから別接続で `KILL QUERY <id>`。最低限、実行中インジケータ横に停止ボタンを。

### U2【高・小工数】タブ操作のショートカットが皆無

- **現状（確認済み）**: `addTab`/`closeTab`/`setActiveTab` は実装済みだが `Cmd+T`/`Cmd+W`/タブ切替/`Cmd+1..9` の**いずれも未バインド**。accelerator は `Cmd+R` のみ。さらに macOS で `Cmd+W` は `menu.ts` の `role:'close'` に奪われ「タブを閉じる」でなく「ウィンドウを閉じる（=接続一覧へ戻る）」に化ける。
- **影響**: 「タブ無制限」を売りにしながらキーボードでタブを開閉・移動できない。
- **v0.3 推奨対応**: メニュー（または renderer keydown）に `Cmd+T`/`Cmd+W`/`Ctrl+Tab`/`Cmd+1..9` を追加。`Cmd+W` は「2枚以上ならタブを閉じる、最後の1枚はウィンドウ動作」に分岐。

### U3【高・中工数】モーダルに Esc クローズもフォーカストラップも無い

- **現状（確認済み）**: `ConnectionFormModal` / `SqlImportModal` のどちらも `role="dialog"`/`aria-modal`/focus-trap/Esc ハンドラが無い。閉じるのは背景クリックとボタンのみ。Tab フォーカスが背後へ抜ける（メモリ `ui-shell-followups` の Esc クローズ後回しが未解決）。
- **v0.3 推奨対応**: 共通 `Modal` コンポーネント化し `role="dialog"` + `aria-modal` + Esc クローズ + フォーカストラップ + フォーカス復帰を一括実装。

### U4【高・中工数】コンテキストメニューが画面端で見切れる + キーボード不可

- **現状（確認済み）**: `ResultsGrid.tsx` / `TableList.tsx` のメニューは `style={{ top: clientY, left: clientX }}` のみで端のフリップ/クランプが無い。下端・右端で項目が画面外に。`role="menu"`/矢印キー/Enter/Esc も無い。**複数のメモリで既知の未解決項目**（`insert-delete-followups` / `table-context-menu-followups` / `connection-groups-followups`）。
- **v0.3 推奨対応**: 共通 `ContextMenu` を作り、(a) ビューポート超過時のクランプ/フリップ、(b) `role="menu"` + 矢印/Enter/Esc。`ResultsGrid` と `TableList` で重複している ctxMenu CSS もここで集約。

### U5〜U9【中〜低】

- **U5 [中]** 非SELECT（INSERT/UPDATE/CREATE 等）の結果が「結果なし（0行）」と出る。「影響行数: N」表示が欲しい（メモリ `ui-shell-followups`）。
- **U6 [中]** エラー表示が生の `code: message` のみ。コピー/再実行ボタン、ローディングのスピナー + 経過秒数が無い。本接続の成否トーストも無い。
- **U7 [中]** 単一セルのコピーが無い（行単位 TSV のみ）。列幅調整・並べ替え・表示/非表示・固定列も無い（メモリ `table-search-jump-followups` のサブC/D）。
- **U8 [中]** INSERT 行のセルに NULL ボタンが無く、空欄（DB デフォルト）と NULL を区別できない（メモリ `insert-delete-followups`）。
- **U9 [低]** 「選択中に Enter で編集開始」未実装、主キーなしテーブルの読み取り専用注記が出ない（メモリ `cell-editing-followups`）。

## 5. アーキテクチャ・技術的負債

## 6. テスト・CI・配布（開発基盤）

> 現状 **`.github/` が無く CI ゼロ、Lint/Format ゼロ、コンポーネントテストゼロ、署名なし配布**。機能追加の前にここを整えると効果が大きい。

- **Q1 [高・小]** GitHub Actions CI を新設。`push`/`pull_request` で `typecheck`→`test`→`build`。`services.mysql`（または compose）で MySQL を立て `TEST_MYSQL_HOST` を渡せば、**現状 default skip されている統合・スキーマテスト計20件が常時実行**に転換。退行検知の費用対効果が最大。
- **Q2 [中・小]** Lint/Format 導入。**Biome 推奨**（lint+format を1ツール・高速・設定最小）。`biome.json` + `lint`/`format` スクリプト + `biome ci` を CI に。`lint-staged` + pre-commit も。
- **Q3 [高・中]** テスト拡充: `useAppStore` の IPC 依存アクション（`runActiveTab`/`commitEdits`/`applyFilters`/`selectTable` 等）は**実質4アクションしかテストされていない**。既存 `useAppStore.connect.test.ts` の `window.api` スタブ手法を流用して追加。さらに `@testing-library/react` + `jsdom` を入れ、vitest を main=node / renderer=jsdom の projects 構成に分離して `ConnectionFormModal`・`FilterBar`・`ResultsGrid` の表示/操作テストを。
- **Q4 [高・中（配布を続けるなら最重要）]** Apple Developer ID 署名 + notarization 導入（electron-builder の `mac.notarize`、Secrets に `APPLE_ID`/`APPLE_TEAM_ID` 等）。現状は ad-hoc 署名で「壊れている」回避が必要・ダブルクリック起動不可。`v*` タグ push で `macos-14` ランナーが署名済み DMG を Release 添付する Release ワークフローも。
- **Q5 [中]** 署名導入後に `electron-updater` + `publish: github` で自動更新（現状は手動 DMG 再取得）。
- **Q6 [低]** Playwright（Electron サポート）で「起動→ホーム表示→モック接続」のスモーク E2E 1本から。

## 8. 推奨 v0.3 スコープ

v0.2.0 で本番接続・SSH・ダンプ入出力という「強い力」を入れた直後なので、v0.3 のテーマは

> **「本番運用の安全性と、大規模データの操作性を固める」**

を提案します。新機能を増やすより、入れた力を安全・快適に使える土台を作る回です。具体的には次の段階構成を推奨します。

### Tier 1 — 安全性（v0.3 の核。これだけは入れたい）
- **S1** 本番ガードを main 側へ（import/破壊的SQLを実ブロック）
- **U1** クエリのキャンセル（停止ボタン + `KILL QUERY`）
- **S2** safeStorage フォールバック是正 + UI 注記
- **S3** 本番CSP + `sandbox: true` + ナビゲーション lockdown
- **S4** ファイルパーミッション `0o600`

### Tier 2 — 基盤と体感速度（並行で着実に効く）
- **Q1/Q2** CI + Biome（最初に入れると以降が安全）
- **P1/P2** 行仮想化 + SQLタブ自動 LIMIT
- **U2** タブ操作ショートカット
- **U3/U4** モーダル・コンテキストメニューの a11y（共通 Modal/ContextMenu）
- **A1/A2/A3** クイックウィン3点（env.d.ts / clearStaging / quoteIdent）

### Tier 3 — 機能拡張（余力に応じて）
- **F2** 外部キージャンプ / **F6** エクスポート形式拡充 / **F9** 設定画面 / **F4** 型対応編集 / **F7** スニペット+タブ復元 / **Q4** 署名+notarization+Release CI

### Tier 4 — 将来（v0.4 以降の大物。設計から）
- **F1** 複数接続 / **A5** slice 分割 / **F3** ビュー・プロシージャ / **F11** PostgreSQL / **F8** 自動再接続・トランザクション

## 付録A. 良くできている点（v0.3 で壊さないこと）

- **SQLインジェクション対策**: 値プレースホルダ・識別子バッククォート2重化・演算子/LIMIT ホワイトリストが一貫（`filterBuilder.ts`/`editBuilder.ts`/`sqlDumpHelpers.ts`）。
- **認証情報の取り扱い設計**: renderer にパスワード・SSH秘匿値を返さない（`stripSecret`、復号は main 内 `getConnectConfig` のみ）。`contextIsolation: true` / `nodeIntegration: false`、preload はホワイトリスト API のみ。
- **import のパス注入防御**: renderer は任意パスを渡せず、main がダイアログで選んだパスを1回だけ消費（`importState`）。FK 無効化を専用接続のセッションで完結。
- **堅牢な状態遷移**: `ApiResult<T>` 判別共用体が全 IPC で一貫、`failTab` で running 固着を回避、編集は BEGIN/COMMIT/ROLLBACK 一括適用、ダンプはストリーミング + バックプレッシャ。
- **ロジック層のユニットテストは厚い**（splitter / importer / filterBuilder / editBuilder / csv 等）。不足は renderer コンポーネントとストア中核アクション。

## 付録B. 監査の方法と限界
